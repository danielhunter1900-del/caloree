How progress photos are kept private
  • 🔑
    Client-side encryption
    Photos are encrypted with AES-256-GCM in the browser before upload. The encryption key is derived from your passphrase (PBKDF2, 600k iterations) and never leaves your device.
  • 🧊
    Server sees ciphertext only
    Supabase Storage holds only encrypted blobs. Even a full database dump reveals no images. Our service role key can't decrypt.
  • 👆
    Device unlock · Face ID / passcode
    The derived key is unwrapped via WebAuthn biometric or a 6-digit PIN. Gallery auto-locks after 60s idle and on tab blur.
  • 🧼
    EXIF + GPS stripped
    Location, camera, and timestamps are stripped from every photo before encryption. No metadata leaks even if a key were compromised.
  • 🌐
    AI never sees photos
    Claude has no read access. Visual comparison is on-device only. AI recaps reference weight/measurements — never image content.
  • 🗑️
    Delete = unrecoverable
    Deleting a photo removes the ciphertext blob. Since we never stored a plaintext copy anywhere, the image cannot be restored.
  • 🛡️
    UK GDPR · your data
    Data-export returns ciphertext + your public metadata. Account deletion wipes the storage bucket. Data is stored in Supabase's EU region (if switchable) or AWS ap-south-1 (current).
Seven rules covering transit, at-rest, access, metadata, and deletion. The key insight: the server never sees plaintext, so GDPR "data subject access" on this subset returns opaque blobs — good for privacy, workable for UK/EU residents.

Implementation outline: new table progress_photos (id, user_id, iso, ciphertext_path, iv, wrapped_key, weight_kg_at_capture, created_at) + a new Supabase Storage bucket progress-photos with bucket-owner-only RLS. Browser does AES-GCM via window.crypto.subtle; PBKDF2 derives the master key from a user passphrase (captured once, never sent). The passphrase is cached in sessionStorage only while unlocked. Add a passphrase-recovery flow via email verification that requires explicit consent (rotating the master key re-encrypts all existing blobs client-side). WebAuthn is used to wrap/unwrap the passphrase on supported devices so users don't re-type every session.
